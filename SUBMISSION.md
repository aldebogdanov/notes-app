# Ответы по тестовому заданию

## 1. Ссылка на форк

<https://github.com/aldebogdanov/notes-app> — всё слито в `main` через серию PR
(по каждой вехе: сначала PR со спецификацией, затем PR с реализацией; история PR
видна в репозитории, план — в `SPEC.md`, детальные спеки — в `docs/specs/`).

## 2. Чем делал

Claude Code (модель Fable 5) в режиме «spec-driven development»: на каждую веху
сначала PR с детальной спецификацией (DDL миграции, схемы, список тестов, риски),
после ревью и merge — PR с реализацией строго по спеке (отклонения фиксируются в той
же спеке и выносятся в описание PR). Между сессиями — Qdrant как память разработки
(решения, грабли); в продуктовый код Qdrant не попадает никак.

## 3. Почему именно эти инструменты

- Такой spec-driven процесс с Claude я уже обкатал на предыдущих задачах — он
  заставляет проговорить дизайн до кода и держит PR компактными и проверяемыми.
- Это моя единственная активная платная AI-подписка на данный момент.
- Связка с Qdrant-памятью уже была проверена именно в этом окружении.
- Только что вышла новая модель (Fable 5) — заодно осознанный тест-драйв.
- Основной опыт работы с AI-инструментами у меня именно с Claude.

## 4. Что заметил в scaffold

- `ruff>=0.5` без верхней границы: свежий 0.15.x форматирует старый код иначе —
  `make lint` красный на чистом checkout (починено в M1; стоит запинить версию).
- Тесты гоняются на SQLite, прод — Postgres: целый класс багов невидим. Поймал
  вживую: `SELECT DISTINCT` по `json`-колонке падает в PG (нет оператора равенства),
  при зелёных тестах.
- Миграции пайтестом не выполняются вообще (фикстуры через `create_all`) —
  `downgrade()` проверяется только руками.
- Нет слоя обратной связи в UI: успешные действия молчат, ошибки — inline-дивы,
  продублированные в пяти страницах, подтверждения — голый `window.confirm`.
  Осознанно не стал чинить (см. п. 5), но это первое, что я бы улучшил.
- Тэги — JSON-массив на заметке, фильтрация по тегу выполняется на Python-стороне
  (диалектно-независимо, но O(n) по заметкам пользователя).
- CI-конфига в репозитории нет, при том что гейты в стиле DoD есть.
- Продуктовый пробел: напоминание умеет приходить только в полночь даты — у
  `note_date` нет компоненты времени. Удобно было бы задавать точное время
  напоминания (на заметку или дефолт на пользователя).

## 5. Что бы сделал следующим заходом

- Ещё адаптеры уведомлений (Slack, Discord, email) — реестр адаптеров и
  `note_notifications.channel` уже спроектированы под это.
- Точное время напоминания (см. наблюдение выше).
- Тосты: `ToastProvider`/`useToast` по образцу существующего `LangProvider`.
- Бейджи статуса напоминания в списке заметок; модальные подтверждения вместо
  `window.confirm`.
- Мульти-воркер: `FOR UPDATE SKIP LOCKED` в шедулере (сейчас один процесс).
- Webhook вместо `getUpdates` для привязки; i18n текста бот-сообщений.
- Запинить ruff, добавить CI, гонять часть тестов на Postgres.

## 6. Known caveats

- Доставка at-least-once: между отправкой и фиксацией статуса есть окно сбоя —
  транзакцией это не лечится (отправка — внешний side effect, у `sendMessage` нет
  ключа идемпотентности). Выбран редкий дубль вместо молча потерянного напоминания.
- Текст бот-сообщений только на английском (вне скоупа UI-i18n).
- Привязка через `getUpdates` ломается, если боту настроить webhook (описано в README).
- Шедулер рассчитан на один процесс backend'а.
- Заметка, созданная позже своей даты (на следующий день и дальше), осознанно
  получает `skipped`; созданная в свой же день — отправляется сразу.
- Демо-скрипт `scripts/demo-reminder.sh` интерактивный (нужно нажать Start в
  Telegram) и ходит в реальный Telegram API — в тестовый контур не включён.
